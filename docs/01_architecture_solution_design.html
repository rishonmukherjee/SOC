1 Project Overview

ComplianceOS is a full-stack compliance management platform addressing two converging regulatory requirements for modern Indian organizations: the SOC 2 Trust Services Criteria framework and the Digital Personal Data Protection (DPDP) Act, 2023.

The platform provides a centralized, role-governed interface for managing organizational risks, security controls, compliance evidence, and data principal rights requests. It maintains a timestamped, append-only activity audit trail for every mutation across all modules.

2 Problem Statement

Organizations pursuing SOC 2 compliance or subject to the DPDP Act face several operational challenges:

  • Compliance tracking is fragmented across spreadsheets, emails, and disconnected tools.
  • There is no formal, auditable workflow for DPDP data principal rights requests such as Access, Erasure, or Correction.
  • Evidence chains linking identified risks to mitigating controls to proof of implementation are not maintained.
  • Data erasure rejections lack documented justifications, exposing organizations to regulatory risk.
  • Role-based access is not enforced, allowing inappropriate actions by non-privileged users.

3 Solution Design

Dashboard

The executive dashboard aggregates real-time metrics: percentage of controls implemented, open risks categorized by severity, evidence artifacts pending review, and DPDP requests with SLA deadlines approaching within 30 days.

Risk Register

Administrators and owners log organizational risks with a title, description, likelihood (1 to 5), and impact (1 to 5). The risk score is calculated automatically by the database as the product of these two values. Risks can be linked to mitigating controls and transition through statuses of Open, Mitigated, and Accepted.

Controls Library

SOC 2 controls are organized by Trust Services Criteria. Controls are assigned to owners and can be linked to each other through typed dependency relationships to represent real-world interdependencies. Supported relationship types are supplements and depends_on.

Evidence Repository

Provides a two-step upload and review workflow. An owner uploads a file or URL as evidence against a specific control. An auditor or administrator then reviews the artifact and transitions its status. If the submission is rejected or needs resubmission, a reason must be documented. Every state change is recorded to the audit log.

DPDP Module

Implements the data principal rights management workflow required under the DPDP Act, 2023. Administrators log incoming requests, each receiving an SLA deadline of 30 days from the received date as specified by the Act. A dedicated Consent Log sub-section tracks consent declarations for specific data processing purposes.

Mandatory Rejection Justification When a DPDP request is rejected, the platform requires the administrator to document a formal legal or operational reason before the status change is saved. This satisfies the audit trail requirement for non-fulfillment of data principal rights.

Activity Logs

A read-only, append-only audit trail capturing every state change across all modules. Entries include entity type, entity ID, action description, actor name, actor ID, and timestamp. Logs are filterable by module.

4 Architecture Overview

ComplianceOS follows a decoupled client-server architecture. The frontend is a single-page React application served from Vercel. The backend is a RESTful FastAPI application served from Render. The two communicate over HTTPS, with the Vercel deployment proxy routing all API requests to the Render service, eliminating cross-origin concerns.

7 Request Lifecycle

A typical state-changing API request follows this path through the system:

  1. The React page calls a function in api.js.
  2. The function attaches X-User-Id and X-Role headers from localStorage and calls fetch against /api/v1/....
  3. Vercel's rewrite rule proxies the request to the Render service URL.
  4. FastAPI's get_current_user dependency resolves the headers into a typed User object.
  5. The route handler performs authorization checks, executes the SQL mutation, and calls log_activity.
  6. The parent handler commits the transaction, atomically persisting both the data change and the audit log entry.
  7. The response is serialized through a Pydantic response model and returned as JSON.

8 Authentication Model

ComplianceOS uses a header-based session model. Upon login, the backend returns a user payload containing the user ID, display name, and role. The frontend stores this payload in localStorage. Every subsequent API call reads this stored payload and attaches it as HTTP request headers.

The backend's get_current_user FastAPI dependency reads these headers on every request and constructs a User object that is injected into route handlers. Route handlers then perform authorization checks against this object.

A quick role switcher in the application header allows instant context switching between the Admin, Owner, and Auditor personas for demonstration purposes, making it straightforward to showcase role-differentiated views and permissions.

9 Key Design Decisions

Transactional Activity Logging

The log_activity helper does not issue a database commit. It writes within the same open database connection as the parent operation. The parent route handler commits both the data change and the audit log entry in a single transaction, guaranteeing that the audit trail is never out of sync with the data it describes. Log write failures are swallowed to prevent a logging issue from rolling back real data.

Mandatory Rejection Documentation

Both the Evidence review and DPDP rejection workflows require a documented reason. This is enforced at the frontend before the API call is made, and the field is stored and returned in all subsequent responses. This creates a durable, retrievable audit record for every rejection decision.

Auditor Restrictions on Erasure Requests

The API layer validates that auditors cannot be assigned as the responsible party for Erasure type DPDP requests. Erasure requests carry the highest legal risk and require a privileged decision-maker. This rule is enforced server-side to prevent bypass through direct API calls.

SQLite Generated Column for Risk Score

The risk score is declared as a SQLite GENERATED ALWAYS AS (likelihood * impact) STORED column. This eliminates any possibility of the score being out of sync with the underlying values, removes the need for application-layer calculation logic, and allows the dashboard to query scores directly in SQL without post-processing.

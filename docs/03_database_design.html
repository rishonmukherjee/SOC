1 Overview

ComplianceOS uses SQLite 3 as its persistence layer. The database consists of 8 normalized tables, designed to capture the full lifecycle of a compliance program including risk identification, control implementation, evidence collection, data principal rights management, and an immutable audit trail.

The schema is defined in server/src/db/schema.sql and initialized on application startup. Demo data is seeded via server/src/db/seed.py, which drops and recreates the database to ensure a clean, consistent state for each deployment.

ID Format All primary keys are human-readable prefixed strings generated at the application layer using the first 6 characters of a UUID hex string. Examples: R-4F2A1B for a risk, E-9C3D7A for an evidence artifact, ACT-88B2F1 for an activity log entry.

4 Table: risks

Stores identified organizational risks. The score column is a SQLite generated column, computed automatically as the product of likelihood and impact.

ColumnTypeKeyConstraint / Default
idTEXTPKNOT NULL
titleTEXTNOT NULL
descriptionTEXTnullable
likelihoodINTEGERNOT NULL, CHECK BETWEEN 1 AND 5
impactINTEGERNOT NULL, CHECK BETWEEN 1 AND 5
scoreINTEGERGENGENERATED ALWAYS AS (likelihood * impact) STORED
statusTEXTDEFAULT 'Open', CHECK IN ('Open', 'Mitigated', 'Accepted')
created_atDATETIMEDEFAULT CURRENT_TIMESTAMP

6 Table: risk_control_links

Junction table establishing the many-to-many relationship between risks and controls. A single risk may be mitigated by multiple controls, and a single control may address multiple risks.

ColumnTypeKeyConstraint
risk_idTEXTPK FKREFERENCES risks(id)
control_idTEXTPK FKREFERENCES controls(id)

The composite primary key (risk_id, control_id) prevents duplicate links.

7 Table: evidence

Stores compliance evidence artifacts submitted by owners and reviewed by auditors. Supports both file uploads (served from the /uploads directory) and URL or text references.

ColumnTypeKeyConstraint / Default
idTEXTPKNOT NULL
control_idTEXTFKNOT NULL, REFERENCES controls(id)
file_url_or_textTEXTNOT NULL
statusTEXTDEFAULT 'Pending Review', CHECK IN ('Pending Review', 'Approved', 'Rejected', 'Needs Resubmission')
uploaded_byTEXTFKREFERENCES users(id)
reviewed_byTEXTFKREFERENCES users(id), nullable
rejection_reasonTEXTnullable, populated when status is Rejected or Needs Resubmission
created_atDATETIMEDEFAULT CURRENT_TIMESTAMP

8 Table: dpdp_requests

Tracks data principal rights requests received under the DPDP Act, 2023. SLA due dates are calculated at the application layer as received_on + 30 days.

ColumnTypeKeyConstraint / Default
idTEXTPKNOT NULL
request_typeTEXTNOT NULL, CHECK IN ('Access', 'Erasure', 'Correction')
data_principal_nameTEXTNOT NULL
statusTEXTDEFAULT 'Open', CHECK IN ('Open', 'In Progress', 'Completed', 'Rejected')
received_onDATENOT NULL
sla_dueDATENOT NULL, set to received_on + 30 days
assigned_toTEXTFKREFERENCES users(id), nullable
rejection_reasonTEXTnullable, required when status is set to Rejected

10 Table: control_dependencies

Self-referential junction table representing typed relationships between controls. Allows modeling of real-world interdependencies such as one control supplementing another or depending on a prerequisite control.

ColumnTypeKeyConstraint
source_control_idTEXTPK FKREFERENCES controls(id)
related_control_idTEXTPK FKREFERENCES controls(id)
relationshipTEXTNOT NULL, CHECK IN ('supplements', 'depends_on', 'replaces')

11 Table: activity_log

Append-only audit trail. Written by the log_activity helper within the same database transaction as the parent operation. Never written directly by route handlers. Never deleted or updated.

ColumnTypeKeyConstraint / Default
idTEXTPKNOT NULL, prefixed ACT-
entity_typeTEXTNOT NULL, e.g. 'Risk', 'Evidence', 'DPDP Request'
entity_idTEXTNOT NULL, references the modified record's ID
actionTEXTNOT NULL, human-readable description
actor_idTEXTnullable, the user who performed the action
actor_nameTEXTnullable, display name captured at write time
timestampDATETIMEDEFAULT CURRENT_TIMESTAMP
Design Note The entity_type and entity_id columns act as a loose, polymorphic foreign key without a strict SQL constraint. This avoids join complexity and allows the log table to reference any entity across any table using a single unified schema.

12 Business Rules and Constraints

  • A dpdp_requests record with status = 'Rejected' must have a non-null rejection_reason. This is enforced at the application layer before the UPDATE is executed.
  • An evidence record with status = 'Rejected' must have a documented rejection_reason. Enforced at the application layer.
  • A user with role = 'auditor' cannot be set as assigned_to for a dpdp_requests record where request_type = 'Erasure'. Validated in the PATCH route handler before the UPDATE executes.
  • The composite primary keys on risk_control_links and control_dependencies prevent duplicate associations at the database level.
  • All CHECK constraints on enumerated TEXT fields (role, status, request_type, etc.) are enforced by SQLite at write time.

13 Risk Severity Classification

The dashboard query classifies open risks into severity tiers based on their computed score. These thresholds correspond to standard industry practice for a 5 by 5 risk matrix where maximum score is 25.

Score RangeSeverity TierRationale
20 to 25CriticalHighest likelihood combined with highest impact
15 to 19HighHigh product of both dimensions
8 to 14MediumModerate risk requiring monitoring
1 to 7LowLow combined likelihood and impact

14 ID Generation Strategy

All record IDs are generated at the application layer using the Python uuid module. The format is a module-specific prefix concatenated with the first 6 hexadecimal characters of a UUID4, converted to uppercase. This provides sufficient uniqueness for demonstration scale while remaining human-readable in the UI and audit logs.
